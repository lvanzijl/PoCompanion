## Duplication Risks

1. **Backlog readiness thresholds are duplicated between domain logic and transport comments.**  
   `PoTool.Core/Health/BacklogStateComputationService.cs` is the executable rule source, while `PoTool.Shared/Health/BacklogStateDtos.cs` repeats the `0/75/100`, `0/25`, and `0/30` semantics in XML comments.

2. **Validation vocabulary is duplicated between Core rule classes and Shared display helpers.**  
   Rule IDs/messages live in Core rule classes, but `PoTool.Shared/WorkItems/ValidationRuleDescriptions.cs` keeps a second manually-maintained mapping of titles and category labels.

3. **Validation interpretation is duplicated outside the validator itself.**  
   `HierarchicalWorkItemValidator` defines the rule outcomes, but `PoTool.Api/Handlers/WorkItems/GetValidationImpactAnalysisQueryHandler.cs` reinterprets parent/child blockage and remediation logic separately instead of consuming a richer domain result.

4. **Effort diagnostics repeat the same aggregation shape across multiple handlers.**  
   `GetEffortImbalanceQueryHandler`, `GetEffortConcentrationRiskQueryHandler`, `GetEffortEstimationQualityQueryHandler`, `GetEffortEstimationSuggestionsQueryHandler`, and `GetSprintCapacityPlanQueryHandler` each:
   - load similar product-scoped work item sets,
   - group by area/iteration/type,
   - compute local thresholds/variance/median heuristics,
   - emit recommendations or warnings.

5. **Sprint/hierarchy progress math remains split inside `SprintTrendProjectionService`.**  
   `ComputeProductSprintProjection`, `ComputeFeatureProgress`, `ComputeEpicProgress`, and `ComputeProgressionDelta` all apply related hierarchy progress semantics in one API service instead of a single CDC-owned progress model.
