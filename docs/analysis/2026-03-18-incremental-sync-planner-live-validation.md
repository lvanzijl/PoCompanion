# Bug Analysis Report

## Metadata
- Bug: Observational Incremental Sync Planner needs live validation
- Area: Sync / planner / validation
- Status: Analysis complete

## ROOT_CAUSE

- The planner is integrated as an observational seam inside `PoTool.Api/Services/Sync/WorkItemSyncStage.cs`, but it does not drive live sync execution yet. `PoTool.Integrations.Tfs/Clients/RealTfsClient.WorkItemsHierarchy.cs` still performs full root-graph discovery and full field hydration for the discovered closure graph, while `PoTool.Api/Services/Sync/WorkItemRelationshipSnapshotStage.cs` and `PoTool.Api/Services/Sync/WorkItemResolutionSyncStage.cs` still rebuild derived data unconditionally. The current planner therefore reports what would be safe to hydrate and rebuild, but live sync behavior is still whole-graph and full-rebuild oriented.
- The current local runtime path fails before any planner output is emitted. During a real staged sync run through `POST /api/CacheSync/1/sync`, `WorkItemSyncStage.BuildIncrementalSyncPlanAsync()` calls `LoadPreviousGraphFactsAsync()`, and its `_context.WorkItems.AsNoTracking().Select(item => new PersistedParentNode(...)).OrderBy(item => item.TfsId)` query in `PoTool.Api/Services/Sync/WorkItemSyncStage.cs` is not translatable by the default SQLite provider. The observed run fetched work items successfully, then failed before `INCREMENTAL_SYNC_PLAN` was logged, so live planner validation is currently blocked by integration rather than by the pure planner contract.
- Even if the stage reaches planner logging, the current `INCREMENTAL_SYNC_PLAN` log only records counts plus reason codes, not the actual `Entered*`, `Left*`, `IdsToHydrate`, or `HierarchyChangedIds` values. The only ID-level validation log is `INCREMENTAL_SYNC_PLAN_VALIDATION`, which reports resolved work items outside the computed closure scope, so live logs are currently too coarse to fully verify per-ID planner correctness.

## CURRENT_BEHAVIOR
- Baseline code inspection shows the pure planner implementation in `PoTool.Core/Sync/DefaultIncrementalSyncPlanner.cs` matches the intended contract in `docs/architecture/incremental-sync-planner.md`: it computes entered/left analytical scope, entered/left closure scope, parent changes within closure, `IdsToHydrate`, and the three rebuild flags deterministically. The existing unit tests in `PoTool.Tests.Unit/Domain/DefaultIncrementalSyncPlannerTests.cs` cover the requested scenarios for a new in-scope item, moved-into-scope reparenting, moved-out-of-scope removal, parent change inside scope, and field-only change.
- The observed staged runtime behavior is different from the intended future contract. A local sync run through the actual API controller and `SyncPipelineRunner` started stage 1, fetched 6,743 work items for ProductOwner 1 from the mock TFS hierarchy, and then failed inside planner-baseline loading before any planner log line was emitted. As a result, the current live pipeline does not yet provide a successful end-to-end planner observation point in the default local environment.
- Newly created item directly inside scope: the pure planner would detect this correctly. A newly discovered child becomes part of both current analytical and closure scope, so it appears in `EnteredAnalyticalScopeIds`, `EnteredClosureScopeIds`, `IdsToHydrate`, and `HierarchyChangedIds`, and it forces relationship and resolution rebuild invalidation. This matches both the contract and `Plan_NewItemInScope_HydratesEnteredItem`.
- Existing item moved into scope by parent change: the pure planner would detect the moved-into-scope bug scenario correctly. Because the item becomes newly reachable from a configured root, it enters current analytical scope and current closure scope even if it previously existed elsewhere. The planner therefore marks it as entered scope and guarantees hydration through entered-closure membership, not only through `ChangedIdsSinceWatermark`. The dedicated unit test `Plan_ItemMovedIntoScope_ByReparenting_IsMarkedAsHierarchyChange` confirms the expected `EnteredAnalyticalScopeIds`, `HierarchyChangedIds`, and `IdsToHydrate` behavior. This means the observational planner logic would have identified the moved-into-scope case correctly once invoked successfully.
- Existing item moved out of scope: the pure planner also models this correctly. An item that was previously reachable but is no longer in current analytical scope is emitted in `LeftAnalyticalScopeIds`, contributes to `HierarchyChangedIds`, does not require hydration, and still forces resolution/projection invalidation. This matches the contract requirement that leaving scope must stay explicit even without a field refresh.
- No-op incremental sync: there is no dedicated unit test, but the implementation is straightforward. If previous and current analytical/closure scope are identical, parent mappings are unchanged, and `ChangedIdsSinceWatermark` contributes no IDs still inside current closure, then all delta sets are empty, `IdsToHydrate` is empty, and all three rebuild flags stay false. That behavior is consistent with the current code, but it has not yet been observed through a successful live staged sync run because the stage currently fails before planning completes.
- Field-only change inside scope: the pure planner behaves correctly here as well. When scope and parent mapping are unchanged but an in-scope ID appears in `ChangedIdsSinceWatermark`, the planner hydrates only that ID, leaves `HierarchyChangedIds` empty, and does not request relationship or resolution rebuilds. `Plan_FieldOnlyChange_HydratesChangedItemWithoutHierarchyInvalidation` verifies this behavior.
- Live sync behavior is still broader than planner output even when the planner is correct. `RealTfsClient.WorkItemsHierarchy` explicitly keeps graph discovery independent from incremental dates and returns the full root-connected graph, while the relationship snapshot and resolution stages rebuild from the full cached state every run. So the current live system is safe against the moved-into-scope bug for a different reason than the planner: it still does full graph retrieval and full downstream rebuilds instead of trusting incremental `IdsToHydrate`.
