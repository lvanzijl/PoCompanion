# Mock Data Quality Audit

_Validated: 2026-03-17_

## Datasets Defined

### 1. Default mock onboarding dataset

Seeded automatically in mock mode through `MockConfigurationSeedHostedService`.

Includes:

- 3 product-owner profiles
- 6 products across those owners
- 8+ linked teams derived from the generated hierarchy
- valid goal-root backlog associations for every product
- an active profile selection so fresh startup lands in a usable state

### 2. Battleship delivery dataset

Generated by the Battleship mock generators.

Includes:

- 10 goals with full objective → epic → feature → PBI/task hierarchy
- 5–10 distinct sprint paths concentrated into a usable recent history window
- pull requests distributed across that sprint history
- dependency and pipeline data with non-trivial variation
- story-point sizing on PBIs only, kept separate from effort-hour fields

## Distributions Used

### Profiles, products, and teams

- multiple product owners are always present in a fresh mock setup
- each owner receives multiple products
- each product receives one or more goal roots
- each product is linked to one or more realistic area-path teams

### Backlog quality

Targeted distribution:

- 80–95% valid active backlog items
- 5–20% invalid active backlog items

Injected invalid patterns:

- missing descriptions on a minority of active epics/features/PBIs
- missing effort estimates on a minority of active epics/features/PBIs
- a small number of features without PBIs
- a separate set of closed-parent / unfinished-descendant state mismatches

### Sprint and history realism

- sprint assignment is concentrated into a rolling 10-sprint mock window instead of a flat 100-sprint spread
- PRs and team-iteration data follow the same recent-history window
- backlog vs sprint allocation remains mixed so views can show both planned and in-flight work

### Effort vs story points

- effort uses hour-style buckets for epic/feature/PBI/bug/task work
- story points are populated on PBIs only
- some PBIs use BusinessValue fallback to exercise canonical story-point resolution
- bugs do not receive story points

## Validation Scenarios Enabled

The upgraded dataset now supports meaningful validation of:

- fresh-clone mock onboarding with no empty profile/product state
- backlog-health variation that is neither all-green nor all-red
- sprint/history trend views with non-flat recent data
- forecast inputs with non-zero story-point signals
- portfolio views with mixed products, teams, inflow, and throughput
- effort diagnostics without reusing story-point values as hours

## Gaps Remaining

- the optional secondary “heavily broken backlog” dataset is not added yet
- the audit currently documents the generated realism rules but does not include screenshot baselines
- PR and pipeline datasets remain single-scenario defaults rather than selectable personas
