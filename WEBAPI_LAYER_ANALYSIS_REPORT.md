### 2.3 Hypotheses for Recurring Build Instability

Based on architectural analysis:

1. **~~Hypothesis A: OpenAPI spec drift~~ (FIXED)**
   - Evidence: Previously two files existed: `swagger.json` (NSwag-generated, OpenAPI 3.0) and `openapi.json` (ASP.NET Core OpenApi, OpenAPI 3.1)
   - Impact: MSBuild target referenced `swagger.json`, nswag.json used `openapi.json`
   - Status: **RESOLVED** - `swagger.json` deleted, single source of truth now

2. **Hypothesis B: Manual regeneration required**
   - Evidence: Client generation runs `BeforeTargets="CoreCompile"` in Debug only
   - Impact: Release builds use stale generated code if not manually regenerated
   - Status: **Design consideration** - regeneration is not deterministic in CI

3. **Hypothesis C: Enum exclusion list maintenance**
   - Evidence: `nswag.json` lines 82-101 list 18 excluded types (enums from Shared)
   - Impact: Adding a new enum to Shared requires updating nswag.json exclusion list
   - Status: **Fragile** - easy to miss when adding enums

## 5. Why Excessive Searching and Scripts

| Task | Current State | Improvement Needed |
|------|---------------|-------------------|

### 5.2 Canonical Workflow Proposal

Create a single entry point: `/tools/regenerate-client.ps1` that:

```powershell
# 1. Start API (generate-openapi.ps1)
# 2. Add operation IDs (add-operation-ids.ps1)
# 3. Fix type issues (fix-openapi-types.ps1)
# 4. Run NSwag (nswag run)
# 5. Verify build
```

**Note**: `regenerate-with-enums.ps1` already exists and does most of this.
